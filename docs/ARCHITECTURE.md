# Architecture

AstraX is structured as a thin browser shell around Qt WebEngine. Qt WebEngine supplies Chromium-based rendering, networking, JavaScript, PDF viewing, and developer tools. AstraX owns the product behavior around that engine.

## Runtime Model

`main.cpp` creates a `QApplication`, parses `--private`, then opens `BrowserWindow`.

`BrowserWindow` owns:

- One `QWebEngineProfile` for the window.
- A `QTabWidget` containing one `BrowserView` per tab.
- Menus, modernized toolbars, shortcuts, status indicators, page information UI, and the Library sidebar.
- `BookmarksStore`, `HistoryStore`, `SettingsStore`, and `SessionStore`.
- A `RequestBlocker` attached to the profile as a URL request interceptor.

`BrowserView` extends `QWebEngineView` only to handle pages that request a new window. When a site opens a target in a new window, the view asks `BrowserWindow` to create a new tab and returns that tab as the destination.

## Persistence

Bookmarks and history are stored in the platform app-data directory selected by `QStandardPaths::AppDataLocation`.

The storage classes use:

- `QJsonDocument` for structured JSON.
- `QSaveFile` for atomic writes.
- URL normalization for duplicate detection.

Private windows still read bookmarks, but they do not append to history. Their profile uses memory cache and no persistent cookies.

Settings and sessions use the same JSON-backed storage pattern. Session restore only keeps HTTP and HTTPS URLs, so internal pages such as `about:blank` are not written back as restorable tabs.

## Request Blocking

`RequestBlocker` subclasses `QWebEngineUrlRequestInterceptor`. It checks each outgoing request against a compact list of common ad/tracker hostnames and path tokens, then blocks matching requests before they reach the network layer. This is intentionally lightweight, but it demonstrates how a browser can enforce policy at the profile boundary.

## UI Responsibilities

The UI layer translates browser actions into Qt WebEngine operations:

- Address bar text goes through `UrlTools::resolveUserInput`.
- Search terms use the configured search URL template.
- The embedded start dashboard is generated by `BrowserWindow` and receives current settings plus bookmark/history counts.
- `BrowserWindow` performs a Qt Network public IP location lookup for live region/country display, then injects the result into the dashboard and falls back to the system locale when offline or unavailable.
- `astrax://search` is handled as an internal page. It ranks bookmarks and history locally, powers live suggestions in the address bar/search page, then exposes the configured web search URL as a fallback.
- AstraX Engine Lab is a separate educational engine module. It includes custom HTML/CSS parsers, DOM inspection, style matching, and block layout calculation without replacing Qt WebEngine.
- Back/forward/reload/stop call the active `BrowserView`.
- Load progress updates the status bar.
- Completed loads update history when not private.
- Downloads are accepted into the user's downloads directory and shown in the Library sidebar.
- Developer Tools are opened as a separate `QWebEngineView`.
- Settings changes update the active request blocker without restarting the app.

## Design Choices

The main AstraX browser intentionally does not replace Chromium's HTML parsing, layout, JavaScript, or TLS for real-world browsing. Those are multi-year browser-engine problems. The product demonstrates production-minded C++ application development around a trusted engine.

The Engine Lab is intentionally scoped differently: it is a learning and debugging surface for browser internals. It now covers HTML parsing, DOM viewing, CSS parsing, selector matching, and basic block layout, then can grow into rendering and JavaScript experiments.
